# Code Review 指南

> 基於 Linux 內核開發經驗制定的代碼審查標準

## 核心原則

### 1. "好品味"優先
- **消除特殊情況**:重寫代碼讓邊界條件變成正常情況
- **簡潔性**:如果需要超過3層縮進,就該重構了
- **一致性**:遵循現有代碼風格,而非個人偏好

### 2. Never Break Userspace
- **API 穩定性**:任何導致現有 API 調用失敗的更改都是 BUG
- **向後兼容**:新功能不能破壞現有功能
- **數據庫 Schema**:必須提供遷移腳本,不能丟失數據

### 3. 實用主義
- **解決實際問題**:代碼要為業務服務,不是為了炫技
- **可維護性**:代碼要讓6個月後的自己能看懂
- **性能考量**:但不要過早優化

## Code Review 檢查清單

### 📋 功能性檢查

#### ✅ 基礎功能
- [ ] **功能完整性**:PR 是否實現了需求中的所有功能點
- [ ] **邊界條件**:空值、最大值、並發訪問等情況的處理
- [ ] **錯誤處理**:異常情況是否有適當的處理和日誌記錄
- [ ] **事務邊界**:`@Transactional` 使用是否正確,回滾邏輯是否合理

#### ✅ API 設計
- [ ] **RESTful 風格**:HTTP 方法和狀態碼使用是否合理
- [ ] **統一響應格式**:是否使用 `ApiResponse<T>` 包裝返回值
- [ ] **參數驗證**:`@Valid` 註解和自定義驗證是否完整
- [ ] **文檔註解**:Swagger 註解是否準確描述了 API 行為

### 🔒 安全性檢查

#### ✅ 認證授權
- [ ] **權限檢查**:需要認證的端點是否正確添加安全註解
- [ ] **SQL 注入**:所有 SQL 查詢是否使用參數化查詢
- [ ] **敏感信息**:密碼、Token 等不能出現在日誌中
- [ ] **輸入驗證**:用戶輸入是否經過適當的驗證和清理

#### ✅ 數據安全
- [ ] **密碼處理**:密碼是否使用 BCrypt 加密存儲
- [ ] **Token 管理**:JWT Token 的生成和驗證邏輯是否正確
- [ ] **數據脫敏**:響應中是否包含不該暴露的敏感字段

### 💾 數據庫相關

#### ✅ SQL 查詢
- [ ] **查詢效率**:是否有 N+1 查詢問題
- [ ] **索引使用**:新的查詢條件是否需要添加索引
- [ ] **分頁查詢**:大數據集是否實現了正確的分頁
- [ ] **併發控制**:是否存在併發更新的數據競爭問題

#### ✅ 數據庫設計
- [ ] **外鍵約束**:實體關聯是否正確定義
- [ ] **數據類型**:金額字段使用 `DECIMAL`,時間字段使用 `TIMESTAMP`
- [ ] **遷移腳本**:Schema 變更是否提供了遷移腳本

### 🏗️ 代碼結構

#### ✅ 架構設計
- [ ] **分層清晰**:Controller/Service/DAO 職責是否分離
- [ ] **依賴注入**:是否正確使用 Spring 的依賴注入
- [ ] **組件註解**:Service 層統一使用 `@Component`,Controller 使用 `@RestController`
- [ ] **接口設計**:Service 層是否抽象出接口(如果有必要)
- [ ] **DTO 使用**:是否正確使用 DTO 而非直接暴露實體

#### ✅ 代碼品質
- [ ] **方法長度**:單個方法是否超過 50 行(需要拆分)
- [ ] **重複代碼**:是否存在可以抽取的公共邏輯
- [ ] **命名規範**:類名、方法名、變量名是否語義明確
- [ ] **註釋品質**:複雜邏輯是否有必要的註釋說明

### 📊 日誌與監控

#### ✅ 日誌規範
- [ ] **結構化日誌**:是否使用 `action=xxx result=xxx` 格式
- [ ] **日誌級別**:INFO/WARN/ERROR 使用是否合理
- [ ] **關鍵操作記錄**:用戶操作、業務流程是否有日誌記錄
- [ ] **性能日誌**:慢查詢、長時間操作是否有監控

#### ✅ 錯誤處理
- [ ] **異常分類**:是否使用合適的異常類型
- [ ] **全局異常處理**:是否通過 `GlobalExceptionHandler` 統一處理
- [ ] **用戶友好**:錯誤信息是否對用戶有意義

### 🧪 測試覆蓋

#### ✅ 測試代碼
- [ ] **單元測試**:核心業務邏輯是否有單元測試
- [ ] **集成測試**:API 端點是否有集成測試
- [ ] **測試數據**:測試是否使用有意義的數據而非硬編碼
- [ ] **測試清理**:測試是否正確清理產生的數據

## Review 流程

### 🔄 標準流程

#### 1. **提交前自檢**
```bash
# 代碼格式化
./mvnw spring-javaformat:apply

# 運行測試
./mvnw test

# 檢查編譯警告
./mvnw compile -Dcompiler.fail.on.warning=true
```

#### 2. **PR 創建規範**
- **標題**: 簡潔描述功能 (例: "添加商品分類樹查詢API")
- **描述**: 包含功能說明、測試範圍、破壞性變更說明
- **自測清單**: 勾選已完成的檢查項目

#### 3. **Review 分配**
- **Senior 開發者**: 負責架構設計和安全性審查
- **Domain 專家**: 負責業務邏輯正確性審查
- **所有人**: 可以參與代碼風格和最佳實踐討論

#### 4. **Review 反饋**
- **LGTM** (Looks Good To Me): 代碼通過審查
- **NACK** (Negative Acknowledgement): 代碼有問題,需要修改
- **討論**: 對實現方式有建議或疑問

### 📝 Review 評論規範

#### ✅ 好的評論示例
```markdown
# 架構問題
NACK: 這個方法違反了單一職責原則,建議拆分成兩個方法:
1. `validateOrder()` - 驗證訂單
2. `processPayment()` - 處理支付

# 安全問題  
NACK: SQL 查詢存在注入風險,請使用參數化查詢:
```sql
-- 當前(危險)
SELECT * FROM users WHERE name = '" + userName + "'
-- 改為
SELECT * FROM users WHERE name = ?
```

# 性能問題
SUGGESTION: 這個查詢可能會導致 N+1 問題,建議使用 JOIN 查詢或者批量查詢

# 代碼風格
NITPICK: 變量命名建議使用更清晰的名稱:`list` -> `activeProductList`
```

#### ❌ 避免的評論
```markdown
# 太模糊
"這裡有問題"

# 太主觀  
"我覺得這樣寫不好"

# 太苛刻
"這是什麼垃圾代碼"
```

## 特定場景檢查

### 🛒 電商業務特有檢查

#### ✅ 庫存管理
- [ ] **庫存扣減**:是否正確處理並發下單的庫存競爭
- [ ] **庫存回滾**:訂單取消時是否正確釋放庫存
- [ ] **負庫存檢查**:是否防止庫存變為負數

#### ✅ 金額計算
- [ ] **精度處理**:金額計算是否使用 `BigDecimal`
- [ ] **舍入規則**:價格計算的舍入方式是否一致
- [ ] **優惠券邏輯**:折扣計算是否正確,是否有最低消費限制

#### ✅ 訂單流程
- [ ] **狀態機**:訂單狀態變更是否遵循正確的業務流程
- [ ] **冪等性**:重複提交是否會創建多個訂單
- [ ] **數據一致性**:訂單創建是否在同一事務中完成

### 🔐 JWT 認證檢查

#### ✅ Token 管理
- [ ] **過期時間**:Access Token 和 Refresh Token 的過期時間是否合理
- [ ] **Token 刷新**:自動刷新邏輯是否正確實現
- [ ] **Token 撤銷**:用戶登出時是否正確撤銷 Token

## 常見反模式

### ❌ 需要立即拒絕的代碼

#### 1. **不安全的代碼**
```java
// 危險:SQL 注入
String sql = "SELECT * FROM users WHERE name = '" + userName + "'";

// 危險:密碼明文日誌
log.info("用戶登入: {} 密碼: {}", username, password);
```

#### 2. **違反業務規則**
```java
// 危險:沒有庫存檢查就扣減庫存
productDao.updateStock(productId, quantity - orderQuantity);

// 危險:金額計算使用 double
double totalPrice = price * quantity;
```

#### 3. **破壞 API 穩定性**
```java
// 危險:刪除現有 API 字段
public class UserResponse {
    // private String email; // 被刪除,會破壞前端
    private String username;
}
```

### ⚠️ 需要討論的代碼

#### 1. **性能問題**
```java
// 警告:可能的 N+1 查詢
for (Order order : orders) {
    List<OrderItem> items = orderItemDao.findByOrderId(order.getId());
}
```

#### 2. **可維護性問題**
```java
// 警告:方法過於複雜,需要拆分
public ApiResponse<OrderResponse> createOrder(...) {
    // 100 行代碼,包含驗證、計算、創建、通知等邏輯
}
```

## 自動化檢查

### 🤖 CI/CD 集成

#### 靜態代碼分析
```yaml
# GitHub Actions 示例
- name: 代碼風格檢查
  run: ./mvnw spring-javaformat:validate

- name: 安全漏洞掃描  
  run: ./mvnw org.owasp:dependency-check-maven:check

- name: 代碼覆蓋率
  run: ./mvnw jacoco:report
```

#### 自動化測試
```bash
# 必須通過的檢查
./mvnw test                    # 單元測試
./mvnw verify                  # 集成測試
./mvnw spotbugs:check         # Bug 檢查
```

## 最後檢查

在 **MERGE** 前,確保:

- [ ] **所有 CI 檢查通過**
- [ ] **至少一位 Senior 開發者 LGTM**
- [ ] **業務邏輯經過相關領域專家確認**
- [ ] **破壞性變更已經充分討論並文檔化**
- [ ] **測試覆蓋了新功能的主要場景**

記住:**代碼質量沒有妥協**。寧願晚發布,也不要帶著技術債務前進。
